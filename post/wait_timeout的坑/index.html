踩坑：wait_timeout

时间： 2020-04-13 分类： 数据库   字数： 945 字 阅读： 2分钟 阅读次数：

(小铭发现并总结)

问题

  1. 早上过来上班，发现在半夜执行的数据统计脚本 统计少了一部分数 值。然后手动再重新执行一遍相同代码后，数据正常了。
  2. 第二天早上过来上班，又发现部分相关的记录根本就 没插入到数据库中。然后又手动再重新执行一遍相同代码后，数据又正常了。
  3. 在同一个脚本内，对于增/删/改=>查（较长时间）=>增/删/改这样的顺序操作，发现第1次的增/删/改可以成功，但第2次的增/删/改却以失败告终。

问题定位

这其实都是由于 mysql 中，一个叫 wait_timeout 的参数设置过低的值而引起的。
此时 mysql 客户端会返回“2013:Lost connection to MySQL server during query”或“2006:MySQL server has gone away”的错误。

关于 wait_timeout

先来看一下wait_timeout参数的定义：
服务器关闭非交互连接之前等待活动的秒数。说人话就是 同一个 mysql 资源句柄下，两个 sql 操作请求之间的 最大时间间隔。
这里有 2 个重点：

  1. 同一个 mysql 句柄：因为 select 操作在不指定库的情况下默认使用的是 slave 库，而 insert/update/delete操作则使用 master 库，两者不属于同一个资源句柄，所以两者之间的时间间隔与 wait_timeout 的设置无关，必须是同一个库的句柄才能遵循 wait_timeout 参数的设置。
  2. 时间间隔：mysql 每执行一次 sql 操作时，相当于是一次 sql 请求，请求完后，mysql 服务器处于空闲状态，直到下一次的 sql 请求到来，而中间的“空闲最大时间”即为 wait_timeout 参数的含义。

问题产生原因分析

问题1：
脚本统计通常是由许多 foreach 中的 select 进行的相加值，如果前一条 select 时间过长，超过了 wait_timeout 的值，那么从第二条 select 开始，都会返回 false，那么累加的值必定会减少。
问题2，3：
数据统计流程的通常是select * n => insert => select * n => insert……在第一次insert成功后，下一次insert操作之前，如果多次 select 操作累加的时间过长（哪怕单次select时间很短），超过了 wait_timeout 的值，那么从第二次开始的 insert 都会返回 false，导致插入数据少了的情况出现 。
备注：之所以有的时候夜间执行不成功，早上过来手动执行成功，是由于夜间统计的长脚本过多，导致sql 操作锁表的情况出现，所以从一定概率上加长了每一条sql的执行时间，从而导致出问题的概率增加。

对于以上问题，只需要把 wait_timeout 参数的值设置高一些即可，如将原来的 60s 设置成 600s。
当然，最重要的还是 sql 语句的优化和业务逻辑的优化。
