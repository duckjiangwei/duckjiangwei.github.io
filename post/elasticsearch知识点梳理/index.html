Elasticsearch 知识梳理

时间： 2022-06-11 分类： 数据库   字数： 3837 字 阅读： 8分钟 阅读次数：

介绍

一个 实时 的 分布式搜索分析 引擎，是一个 非关系型 数据库

特性

  1. 高效聚合：支持各种聚合方式
  2. 全文检索，分词
  3. 相关性评分
  4. 集群：天然支持集群，横向扩展简单

概念

文档元数据

一个文档不仅仅包含它的数据 ，也包含 元数据。3 个必须的元数据如下：

  • _index：文档存在哪里，类似于 mysql 里的表
  • _type：文档表示的对象类别。在新版本的 es 里，一个索引里只能一个 type
  • _id：文档唯一标识

还有一些其他的元数据。

文档更新

在 es 中，文档是 不能 被修改的，只能被替换：

  • 无论是使用 index API 还是 update API，其基本逻辑都是：从旧文档构建 JSON=>更改该 JSON=>删除旧文档=>索引一个新文档。
  • 但 index API 其底层是通过 get 先获取原有文档，再重新 index 一个新文档，所以一次操作用到了 get 和 index 请求；而对于 update API 的修改文档操作则是在分片内部完成的，这样就避免了多次请求的网络开销。并且通过减少“检索和重建索引”这两个步骤之间的时间（因为在一个分片内完成，避免多次请求网络开销），我们也减少了其他进程的变更带来冲突的可能性。但尽管如此，update API 还是有可能发生更新冲突（此时会更新失败），这是可以通过设置 url 参数 retry_on_conflict=n 来使在发生更新冲突进行 n 次重试。

旧的文档不可见，会在 es 空闲的时候删除。

并发控制

有 2 种：

  • 悲观并发控制：它假定有变更冲突可能发生，典型的例子是读取一行数据之前先将其锁住，确保只有放置锁的线程能够对这行数据进行修改。关系型数据库，比如 mysql 就这么干
  • 乐观并发控制：先假定冲突是不可能发生，然而，如果源数据在读写当中被修改，更新将会失败。程序接下来将决定该如何解决冲突：重试更新、使用新的数据、或者将相关情况报告给用户。es 采用这种

es 是分布式的，当文档创建、更新或删除时， 新版本的文档必须复制到集群中的其他节点。
es 也是异步和并发的，这意味着这些复制请求被并行发送，并且到达目的地时也许 顺序是乱的 。
故 es 需要一种方法确保文档的旧版本不会覆盖新的版本。

es 根据 version 的值实现乐观并发控制，保证在多个线程并行修改一条数据时不出问题。

路由文档到分片

es 如何知道一个文档应该存放到哪个分片中呢？
根据如下公式：

# number_of_primary_shards 是分片的数量，routing 默认是文档_id，也可自定义
shard = hash(routing) % number_of_primary_shards

这就解释了为什么我们 要在创建索引的时候就确定好主分片的数量 并且永远不会改变这个数量：因为如果数量变化了，那么所有之前路由的值都会无效，文档也再也找不到了。

routing 可以自定义，但是这样有个问题：比如用 uid 作为 routing，当某个用户的请求特别多，会出现数据偏移，意味着分片存储的文档数量不均，个别分片过于臃肿，对查询效率造成影响。

主分片和副分片

es 是个分布式搜索引擎，索引通常都会分解成多个不同的部分，这些分布在不同节点的数据就是 分片。
es 默认创建 5 个主分片，同时为每个主分片创建 1 个副分片。

主副分片 都能处理查询请求，但只有主分片能处理「索引，删除」等写请求。因为数据是从主分片同步到副分片。
注意：这一点和 kafka 不同，kafka 的副分片仅仅做数据的备份，不参与读和写。

因为路由寻址的关系，对于已经存在的索引，只能动态改副本数量，不能改主分片数量。

经验：一个分片最多存储的理论文档数为 2,147,483,519 个，由于创建索引的时候分片数已经固定，所以一个索引所能存储的数据量是固定的。实际使用过程中应根据实际索引的数据量来设置分片数。数据量一定的情况下，分片数过少使得分片不能平均分布在节点上以并行 IO 提高搜索性能；分片数过多就需要每个分片都查询一次，然后聚合结果，又会造成开销过大。

自动扩容：
当集群规模扩大或者缩小时， es 会自动的在各节点中迁移分片，使得数据仍然均匀分布在集群里。

es 增删改查流程

新建，索引和删除文档

  1. 客户端向 Node 1 发送新建、索引或者删除请求。
  2. 节点使用文档的 _id 确定文档属于分片 0 。请求会被转发到 Node 3，因为分片 0 的主分片目前被分配在 Node 3 上。
  3. Node 3 在主分片上面执行请求。如果成功了，它将请求并行转发到 Node 1 和 Node 2 的副本分片上。一旦 所有 的副本分片都报告成功, Node 3 将向协调节点报告成功，协调节点向客户端报告成功。

取回文档

  1. 客户端向 Node 1 发送获取请求
  2. 节点使用文档的 _id 来确定文档属于分片 0 。分片 0 的副本分片存在于所有的三个节点上。 在这种情况下，它将请求转发到 Node 2 。
  3. Node 2 将文档返回给 Node 1 ，然后将文档返回给客户端。

注意：在处理读取请求时，协调结点在每次请求的时候都会通过 轮询所有的副本分片来达到负载均衡。

局部更新文档

  1. 客户端向 Node 1 发送更新请求
  2. 它将请求转发到主分片所在的 Node 3 。
  3. Node 3 从主分片检索文档，修改 _source 字段中的 JSON ，并且尝试重新索引主分片的文档。 如果文档已经被另一个进程修改，它会重试步骤 3 ，超过 retry_on_conflict 次后放弃。
  4. 如果 Node 3 成功地更新文档，它将 新版本的文档 并行转发到 Node 1 和 Node 2 上的副本分片，重新建立索引。 一旦所有副本分片都返回成功， Node 3 向协调节点也返回成功，协调节点向客户端返回成功。

注意：基于文档的复制：当主分片把更改转发到副本分片时， 它不会转发更新请求。 相反，它转发完整文档的新版本。请记住，这些更改将会异步转发到副本分片，并且不能保证它们以发送它们相同的顺序到达。 如果 es 仅转发更改请求，则可能以错误的顺序应用更改，导致得到损坏的文档。

多文档模式

mget 取数据

  1. 客户端向 Node 1 发送 mget 请求。
  2. Node 1 为每个分片构建多文档获取请求，然后 并行转发这些请求 到托管在每个所需的主分片或者副本分片的节点上。一旦收到所有答复， Node 1 构建响应并将其返回给客户端。
bulk 修改多个文档

  1. 客户端向 Node 1 发送 bulk 请求。
  2. Node 1 为每个节点创建一个批量请求，并 将这些请求并行转发 到每个包含主分片的节点主机。
  3. 主分片一个接一个按顺序执行每个操作。当每个操作成功时，主分片并行转发新文档（或删除）到副本分片，然后执行下一个操作。 一旦所有的副本分片报告所有操作成功，该节点将向协调节点报告成功，协调节点将这些响应收集整理并返回给客户端。

问题：客户端请求的是 Node 1，它怎么知道有几个节点？同时，各个节点可能在不同的服务器上，它怎么知道地址？
es 集群中的 每个节点都会存储集群状态（slave 节点只负责接收从 master 同步过来的集群信息而没有维护的权利），知道索引内各分片所在的节点位置，因此在整个集群中的任意节点（此处称为“协调节点”）都可以知道一条数据该往哪个节点分片上存储。反之也知道该去哪个分片读。所以，es 不需要将读写请求发送到 master 节点，任何节点都可以作为数据读写的切入点对请求进行响应。这样进一步减轻了 master 节点的网络压力，同时提高了集群的整体路由性能。
这点和 kafka 里的 ZooKeeper 差不多，它是一个分布式协调框架，负责协调管理并保存 Kafka 集群的所有元数据信息，比如集群都有哪些 Broker 在运行、创建了哪些 Topic，每个 Topic 都有多少分区以及这些分区的 Leader 副本都在哪些机器上等信息。

倒排索引

概念：es 将文本的内容拆分成单独的词（词条或者叫tokens），这个动作即 分词 ，再经过 标准化 后，创建 一个包含所有不重复词条的排序列表，然后列出每个词条出现在哪个文档。

标准化 规则：

  • 转化为小写：Quick 可以小写化为 quick
  • foxes 可以 词干提取 –变为词根的格式– 为 fox 。类似的， dogs 可以为提取为 dog 。
  • jumped 和 leap 是同义词，可以索引为相同的单词 jump 。

倒排索引作用：通过建立 分词=>文档 的关系，提升搜索速度，同时，也利于相关性排序。

空集群

如果我们启动了一个单独的节点，里面不包含任何的数据和索引，就是一个包含空内容节点的集群了。

一个运行中的 es 实例称为一个节点，而集群是由一个或者多个拥有相同 cluster.name 配置的节点组成， 它们共同承担数据和负载的压力。当有节点加入集群中或者从集群中移除节点时，集群将会重新平均分布所有的数据。

当一个节点被选举成为 主 节点时， 它将负责管理集群范围内的所有变更，例如增加、删除索引，或者增加、删除节点等。
而主节点并不需要涉及到文档级别的变更和搜索等操作，所以当集群只拥有一个主节点的情况下，即使流量的增加它也不会成为瓶颈。 任何节点都可以成为主节点。我们的示例集群就只有一个节点，所以它同时也成为了主节点。

故障转移

启动第二个节点

当你在同一台机器上启动了第二个节点时，只要它和第一个节点有同样的 cluster.name 配置，它就会自动发现集群并加入到其中。 但是在不同机器上启动节点的时候，为了加入到同一集群，你需要配置一个可连接到的单播主机列表。 详细信息请查看 最好使用单播代替组播 。

当第二个节点加入集群后，集群将会重新平均分布所有的数据。第二个节点也拥有全量的数据，当其中一个节点挂了，依旧可以提供服务。

问题：如果我们要扩容到超过 6 个节点，怎么办？
主分片的数目在索引创建时就已经确定了下来。实际上，这个数目定义了这个索引能够 存储 的最大数据量。（实际大小取决于你的数据、硬件和使用场景。） 但是，读操作——搜索和返回数据——可以同时被主分片 或 副本分片所处理，所以当你拥有越多的副本分片时，也将拥有越高的吞吐量。

分布式

Elasticsearch 天生就是分布式的，它尽可能地屏蔽了分布式系统的复杂性。
这里列举了一些在后台自动执行的操作：

  • 分配文档到不同的容器 或 分片 中，文档可以储存在一个或多个节点中
  • 按集群节点来均衡分配这些分片，从而对 索引和搜索过程进行负载均衡
  • 复制每个分片以 支持数据冗余，从而防止硬件故障导致的数据丢失
  • 将集群中任一节点的请求路由到存有相关数据的节点。(路由寻址)
