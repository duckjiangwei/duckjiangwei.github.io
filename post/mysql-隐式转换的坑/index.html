MySQL:隐式转换的坑

时间： 2021-04-08 分类： 数据库   字数： 812 字 阅读： 2分钟 阅读次数：

隐式类型转换

首先建表如下，注意字段的类型：

mysql> CREATE TABLE `tradelog` (
  `id` int(11) NOT NULL,
  `tradeid` varchar(32) DEFAULT NULL,
  `operator` int(11) DEFAULT NULL,
  `t_modified` datetime DEFAULT NULL,
  PRIMARY KEY (`id`),
  KEY `tradeid` (`tradeid`),
  KEY `t_modified` (`t_modified`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

现有查询如下：

mysql> select * from tradelog where tradeid=110717;

explain 分析后发现，这条 sql 走了全表扫描。因为 tradeid 的类型是字符串，而输入的参数是整形，所以需要做类型转换。
这里会有 2 给问题：

  1. 数据类型转换的规则是什么
  2. 为什么发生了类型转换，就需要走全索引扫描

你可以执行 select "10" >9：

  1. 如果规则是“将字符串转成数字”，那么就是做数字比较，结果应该是 1；
  2. 如果规则是“将数字转成字符串”，那么就是做字符串比较，结果应该是 0。

select "10" > 9的结果为 1，所以：
隐式转换规则一：字符串和数字做比较的话，是将字符串转换成数字
也就是说：

mysql> select * from tradelog where tradeid=110717;
# 等价于

mysql> select * from tradelog where  CAST(tradid AS signed int) = 110717;

因为 对索引字段做了函数操作，所以没法用上索引。

隐式字符编码转换

再加一个表：


mysql> CREATE TABLE `trade_detail` (
  `id` int(11) NOT NULL,
  `tradeid` varchar(32) DEFAULT NULL,
  `trade_step` int(11) DEFAULT NULL, /*操作步骤*/
  `step_info` varchar(32) DEFAULT NULL, /*步骤信息*/
  PRIMARY KEY (`id`),
  KEY `tradeid` (`tradeid`)
) ENGINE=InnoDB DEFAULT CHARSET=utf8;

insert into tradelog values(1, 'aaaaaaaa', 1000, now());
insert into tradelog values(2, 'aaaaaaab', 1000, now());
insert into tradelog values(3, 'aaaaaaac', 1000, now());

insert into trade_detail values(1, 'aaaaaaaa', 1, 'add');
insert into trade_detail values(2, 'aaaaaaaa', 2, 'update');
insert into trade_detail values(3, 'aaaaaaaa', 3, 'commit');
insert into trade_detail values(4, 'aaaaaaab', 1, 'add');
insert into trade_detail values(5, 'aaaaaaab', 2, 'update');
insert into trade_detail values(6, 'aaaaaaab', 3, 'update again');
insert into trade_detail values(7, 'aaaaaaab', 4, 'commit');
insert into trade_detail values(8, 'aaaaaaac', 1, 'add');
insert into trade_detail values(9, 'aaaaaaac', 2, 'update');
insert into trade_detail values(10, 'aaaaaaac', 3, 'update again');
insert into trade_detail values(11, 'aaaaaaac', 4, 'commit');

此时有 sql 语句如下：


mysql> select d.* from tradelog l, trade_detail d where d.tradeid=l.tradeid and l.id=2; /*语句Q1*/

根据 join 的查询原理，这个 sql 执行的步骤应该为：

  1. 先在 tradelog 上查到 id=2 的行，此时能用上主键
  2. 接着循环用结果的 tradeid 去 trade_detail 表找

如果你 explain 的话，会发现步骤2没用上索引，如果单独把这一步改成 sql 语句的话：

mysql> select * from trade_detail where tradeid=$L2.tradeid.value;

其中 $L2.tradeid.value 的字符集是 utf8mb4，和 trade_detail 的 tradeid 并不相同。因为字符集 utf8mb4 是 utf8 的超集，所以当这两个类型的字符串在做比较的时候，MySQL 内部的操作是，先把 utf8 字符串转成 utf8mb4 字符集，再做比较。
所以等价于：

select * from trade_detail  where CONVERT(traideid USING utf8mb4)=$L2.tradeid.value;

很明显，又对字段加了函数，用不上索引。

注意，如果 trade_detail 表是 utf8mb4 ,而 tradelog 表是 utf8，那么等价于:

select operator from tradelog  where traideid =CONVERT($R4.tradeid.value USING utf8mb4);

此时就可以用上索引。
隐式转换规则二：字符编码不同时，会将范围小的字符编码转换成它的超集
