缓存问题

时间： 2019-08-30 分类： redis   字数： 955 字 阅读： 2分钟 阅读次数：

缓存不一致

原因：
数据库和缓存里要同步更新，有时候其中一个的更新失败。

缓存雪崩

原因：

  1. 大量缓存同时过期
  2. redis 缓存实例宕机

方案：

  1. 缓存过期时间上下浮动几分钟
  2. 服务降级：当用户访问的是非核心数据，当读缓存失效，直接返回空，不走数据库；当访问的是核心数据，才直接走数据库
  3. 服务熔断：当雪崩产生时，暂停对缓存系统的访问，直接返回空，也不走数据库。
  4. 请求限流：限制每秒运行的请求数，减轻数据库的压力。

判断缓存雪崩：
可以监测 Redis 缓存所在机器和数据库所在机器的负载指标，例如每秒请求数、CPU 利用率、内存利用率等。如果我们发现 Redis 缓存实例宕机了，而数据库所在机器的负载压力突然增加（例如每秒请求数激增），此时，就发生缓存雪崩了。

缓存击穿

原因：
针对 某个 访问非常频繁的热点数据的请求，无法在缓存中进行处理，紧接着，访问该数据的大量请求，一下子都发送到了后端数据库。

击穿和雪崩的区别：
雪崩是多个缓存失效，击穿的某个热点数据的缓存失效。

方案：

  1. 热点数据的缓存不给过期时间

缓存穿透

原因：
要访问的数据既不在 Redis 缓存中，也不在数据库中，导致请求在访问缓存 时，发生缓存缺失，再去访问数据库时，发现数据库中也没有要访问的数据。

方案：

  1. 缓存空值或者缺省值。比如查 id = 999 的用户没查到，可以在缓存里加个 999=>0 的缓存。
  2. 使用布隆过滤器，快速判断数据是否存在
  3. 请求检测：对系统接收的数据进行合法性检测，过滤掉恶意请求

布隆过滤器的原理：

  1. 弄一个初始值都为 0 的数组。弄 N 个哈希函数
  2. 当数据写入数据库的时候，布隆过滤器通过 3 个操作完成标记：
    1. 使用 N 个哈希函数，分别计算这个数据的哈希值，得到 N 个哈希值
    2. 把这个 N 个 hash 值对数组的长度取模，值即为数组中的位置
    3. 把对应的位置的值置 1

当查询某个值的时候，执行上述过程，如果这 N 个位置有一个值不为 1 ，则数据一定不存在。
当然了，也可能全为 1， 数据也不存在的。因为别的数据哈希后的 N 个值可能也占这个位置。

布隆过滤器的特点：

  1. 空间效率和查询效率高。
  2. 不存在漏报，即某个元素在某个集合中，肯定能报出来。
  3. 可能存在误报，即某个元素不在某个集合中，可能也被爆出来，随着存入的元素数量增加，误算率随之增加。
  4. 不支持删除。如果想删除，可以参考下 布谷鸟过滤器

布隆过滤器里的那个数组长度可以特别大，可以亿级别的。如果存我们的用户量，也是够用了。
同时，还可以弄多个布隆过滤器，定期 全量同步 切换。
